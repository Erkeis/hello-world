## 🏛️ 1. Architecture: The Layered Skill Model
Skills and automation tools must be categorized by their **Abstraction Layer** to prevent "Project Pollution" during export.

- **Layer 1: Global/Root (Management Domain)**
    - *Purpose:* Infrastructure maintenance, version control monitoring, session purity.
    - *Persistence:* These remain in the project root or user's global skill directory. They are "Management Harnesses."
    - *Example:* `Git-Harness Sanity Checker`, `Agent Lifecycle Manager`.
- **Layer 2: Application-Specific (Product Domain)**
    - *Purpose:* Business logic, specialized HR flows for the Dashboard UI.
    - *Persistence:* These are candidates for `/exprj` (Export). They define the *behavior* of the product being built.
    - *Example:* `Agile HR Dispatcher`, `UI Component Generator`.

## 🛠️ 2. Methodology: Strict Handover Hygiene
When preparing for a project handover or export (`/exprj`), the following routine must be followed:

1.  **Meta-Description Enforcement:** Every skill MUST have a `metadata` section defining its layer. If it lacks a clear scope, it is "Technical Debt."
2.  **The .exportignore Strategy:** Use `.exportignore` to explicitly mark files that are essential for *development* (Layer 1) but should not pollute the *final product* (Layer 2).
3.  **Context Lean-ness:** Before export, run a "De-cluttering" script to remove temporary logs, blocked flags, and session-specific metadata.

## 🛡️ 3. Rutine: "Copy-on-Export" Pattern
To ensure the parent environment retains its management capability while the child (export) gets its functionality:
- **Strategy:** Instead of moving assets, use a script-driven copy approach where common tools are replicated in the project root during export, while app-specific ones are packaged into the export artifact.
