## 🏛️ 1. Architecture: Factory vs. Showroom Pattern
- **Source Isolation (Factory):** Repositories containing multiple skills (like `gstack`) should reside in `~/.gemini/extensions/<name>/`. This avoids duplicate discovery conflicts.
- **Runtime Activation (Showroom):** Individual skill definitions should be symlinked or placed separately into `~/.gemini/skills/gstack-<feature>/`. 
- **Conflict Root Cause:** Gemini CLI scans `~/.gemini/skills/` recursively. Placing a full repository inside this folder causes the root `SKILL.md` to collide with individual feature `SKILL.md` files (Recursive Collision).

## 🛠️ 2. Tooling: Porting Claude-Code to Gemini CLI
- **Host Detection Mapping:** Many 2026-era tools (gstack, etc.) use a `--host` flag. If `gemini` is not explicitly supported, use `auto` or check for `.agents/skills/` output which follows the Gemini-compatible Markdown spec.
- **Standard Registration:** Adding new extensions to `~/.gemini/extensions/extension-enablement.json` is required for proper system-wide recognition and discovery.

## 🛡️ 3. OS Pitfalls: Windows-Specific Environment Issues
- **Silent Script Failures (CRLF):** `git clone` on Windows often introduces `\r\n` line endings. Bash scripts will fail with cryptic errors (e.g., `$'\r': command not found`). 
  - *Mitigation:* Use `sed -i 's/\r//g' <file>` within a bash context before execution.
- **PowerShell Wildcard Trap:** `Move-Item -Path source\* -Destination dest` can behave inconsistently with hidden files or deeply nested structures. 
  - *Golden Rule:* Always verify destination count before deleting source `temp` directories.

### Bash Line-Ending Fix (Batch)
