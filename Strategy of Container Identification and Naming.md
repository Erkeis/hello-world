# 컨테이너 식별 및 네이밍 전략

## 문제 정의

각 container가 고유한 미션을 고정적으로 갖는 것이 아니라, agile하게 업무를 처리하고 세션마다 새로운 역할을 부여받는 구조라면 다음 질문이 생깁니다.

- 각 container에 설정된 `container id`, `port num`, `역할`만으로 충분한가?
- 아니면 고유한 이름을 두는 편이 더 나은가?
- 오케스트레이터가 `port num`과 `name`을 모두 받는 편이 실수를 줄이는가?
- Git 식별은 container별 고유 account가 나은가, 아니면 tag/prefix 같은 별도 규칙이 더 효율적인가?

## 결론

핵심 결론은 세 가지입니다.

1. 컨테이너에는 고정된 이름을 두는 편이 낫습니다.
2. 오케스트레이터에는 `port`와 `name`을 함께 전달하는 편이 안전합니다.
3. Git 식별은 `공용 계정 + prefix/tag 컨벤션`이 agile한 운영에 더 잘 맞습니다.

## 전략 다이어그램 텍스트 버전

```text
Orchestrator Hub
(CEO/PM · context 수신 측)
|
+-- 전략 A: Port + ID만
|   예: container-b:8002
|   - 역할은 매 세션 주입
|   - 단점: git log 판독 비용 높음
|   - 단점: 역할 혼동 가능
|
+-- 전략 B: 고정 이름 부여
|   예: sec-forge:8002
|   - 이름 = 정체성 고정
|   - 장점: git author 즉각 인식
|   - 장점: 역할 귀속 명확
|
+-- 전략 C (권장): 혼합 레이어
    예: sec-forge:8002 [sec-lead]
    - 이름 + 포트 + 세션 역할
    - 장점: 이름으로 불변 정체성 유지
    - 장점: 역할은 유연하게 재배정 가능
```

## Git 식별 계층 비교

| 방식 | 예시 | 장점 | 단점 | 적합성 |
| --- | --- | --- | --- | --- |
| 개별 Git Account | `sec-forge@team` / `qa-scout@team` | 감사 추적이 강함 | 설정 비용이 높고, 역할 재배정 시 account 정체성과 실제 역할이 어긋날 수 있음 | 고정 역할 조직 |
| 공용 계정 + 태그/컨벤션 | `agent/sec-forge: fix auth` | 설정 단순, 역할 재배치 유연, 오케스트레이터가 prefix로 즉시 판독 가능 | 규칙이 느슨하면 로그 품질이 떨어질 수 있음 | agile 구조에 적합 |

## 왜 고정 이름이 필요한가

`container-b:8002`는 오케스트레이터가 매번 역할 매핑을 다시 조회해야 합니다. 반면 `sec-forge:8002`는 이름 자체가 1차 의미를 담기 때문에, 컨텍스트를 읽는 인지 비용이 크게 줄어듭니다.

즉, 역할이 세션마다 바뀌더라도 컨테이너의 기본 성격은 이름에 고정해두는 편이 좋습니다. 이는 사람 조직에서 "직책"과 "정체성"을 구분하는 방식과 비슷합니다.

## 왜 Port와 Name을 함께 넘겨야 하는가

오케스트레이터 입장에서 두 값의 역할은 다릅니다.

- `port`: 라우팅과 연결용 식별자
- `name`: 의미 해석과 컨텍스트 파악용 식별자

둘 중 하나만 쓰면 실수가 늘어날 수 있습니다. 가장 실용적인 형태는 다음과 같습니다.

```text
sec-forge:8002 [sec-lead]
```

이 표현은 다음 세 층을 분리합니다.

- `sec-forge`: 불변의 정체성
- `8002`: 시스템 라우팅 정보
- `sec-lead`: 세션별 가변 역할

## Git 식별 전략에 대한 판단

Git에서는 `누가 커밋했는가`보다 `어떤 에이전트 정체성으로 작업했는가`를 빠르게 파악하는 것이 더 중요할 수 있습니다. 특히 역할이 자주 바뀌는 구조라면, container별 고정 account는 오히려 혼선을 만들 수 있습니다.

예를 들어 `qa-scout` 계정이 보안 관련 커밋을 남기면, 실제 세션 역할과 계정 이름이 어긋나 해석 비용이 커집니다.

반대로 아래처럼 commit prefix를 쓰면 의미 계층이 깔끔해집니다.

```text
agent/sec-forge: fix auth flow
agent/sec-forge[sec-lead]: review token broker
```

이 방식의 장점은 다음과 같습니다.

- author 파싱 없이 prefix만으로 즉시 귀속을 파악할 수 있습니다.
- 역할이 바뀌어도 prefix 규칙만 유지하면 로그 해석이 쉽습니다.
- 계정 수를 늘리지 않아도 되어 운영 부담이 줄어듭니다.

## 권장안

가장 균형이 좋은 선택은 다음과 같습니다.

- 컨테이너는 고정 이름을 가집니다.
- 오케스트레이터는 `name + port`를 함께 받습니다.
- 세션 역할은 별도 필드나 bracket 표기로 덧붙입니다.
- Git은 `공용 계정 + prefix/tag 컨벤션`으로 운영합니다.

예시:

```text
container name: sec-forge
port: 8002
session role: sec-lead
git prefix: agent/sec-forge[sec-lead]
```

## 한 줄 요약

이름은 컨테이너의 불변 정체성으로, 역할은 세션의 가변 정체성으로 분리하는 것이 핵심입니다. 이 구조가 오케스트레이터의 인지 부하를 줄이고, Git 로그 해석과 운영 유연성도 함께 높여줍니다.
